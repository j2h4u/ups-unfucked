# Phase 12.1: Math Kernel Extraction & Formula Stability Tests - Research

**Researched:** 2026-03-16
**Domain:** Battery formula system refactoring, simulation architecture, mathematical stability validation
**Confidence:** HIGH

## Summary

Phase 12.1 extracts all existing battery mathematics (Peukert, SoH calculation, SoC interpolation, calibration) into a pure-function kernel package (`src/battery_math/`), then validates mathematical stability through a year-long simulation harness with per-iteration Lyapunov tests, permutation tests, and adversarial scenarios. This is a prerequisite gate: Phase 12 (capacity estimation) cannot proceed safely until the formula chain proves stable under synthetic stress. The phase establishes an architectural invariant: the kernel knows nothing about daemon I/O, the daemon knows nothing about math, and both orchestrators (daemon + simulator) treat the kernel as a pure function library.

**Primary recommendation:** Extract math into focused modules by change frequency (Peukert stable ~forever, SoH statistical ~weekly). Use a frozen `BatteryState` dataclass to make circular dependencies structurally visible. Build simulation to iterate pure functions with synthetic discharge events; validate per-iteration divergence ratio < 1.0 (not aggregate ±3% test).

- **Tightened discharge-mode voltage floor** 10.0V during OB (vs. global 8.0V) — rejects physically impossible UPS voltage states.
- **Raw load in discharge buffer** (not EMA-filtered) for coulomb counting accuracy.

## Architecture Patterns

### Pattern 1: Math Kernel as Focused Module Package

**What:** Extract formulas into `src/battery_math/` with one file per change-frequency domain (Peukert stable, SoH evolving, capacity new).

**When to use:** Any codebase with mixed-frequency formula updates. Prevents "every tweak touches foundational physics" anti-pattern.

**Structure:**
```
src/battery_math/
    __init__.py                 # Re-exports: from .peukert import *; from .soh import *
    peukert.py                  # peukert_runtime_hours(load, capacity, exp, ...) → float
    soh.py                      # calculate_soh_from_discharge(...) → float | None
    soc.py                      # soc_from_voltage(voltage, lut) → float
    calibration.py              # calibrate_peukert(...) → float | None
    capacity.py                 # estimate_capacity(...) → (float, float, dict) [Phase 12]
    types.py                    # BatteryState frozen dataclass
```

### Pattern 2: Frozen State Dataclass for Type Safety

**What:** Kernel functions take immutable `BatteryState(frozen=True)` and return new `BatteryState` — not mutate-in-place.

**Why:** Circular dependencies become structurally visible. `calculate_soh(battery_state) → BatteryState` shows at the type level what goes in, what comes out. Hidden state mutations hide bugs.

**Example:**
```python
from dataclasses import dataclass

@dataclass(frozen=True)
class BatteryState:
    """Immutable battery state snapshot. Every kernel function takes/returns this."""
    soh: float                          # State of Health [0.0, 1.0]
    peukert_exponent: float             # Peukert exponent [1.0, 1.4]
    capacity_ah_rated: float            # Rated capacity (Ah) from manufacturer label
    capacity_ah_measured: float | None  # Measured capacity once converged (Phase 12 result)
    lut: tuple[tuple[float, float, str], ...]  # Voltage→SoC LUT, immutable
    cycle_count: int                    # Number of OL→OB transitions
    cumulative_on_battery_sec: float    # Total OB duration (seconds)
```

Kernel function signature:
```python

    battery_state: BatteryState,  # Input immutable state

    discharge_time_series: List[float],
    # ... other args ...
) -> BatteryState | None:  # Returns new state or None if invalid
    # Never mutate battery_state or its fields
    # Return new BatteryState(...) with updated soh field
    pass
```

**When to use:** Separating algorithmic math from operational policy.

### Anti-Patterns to Avoid

1. **Don't call `time.time()` inside kernel functions** — breaks simulation purity. Inject as parameter.
2. **Don't mutate BatteryState or LUT inside kernel functions** — return new state. Frozen dataclass prevents accidents.
3. **Don't mix logging/I/O with math functions** — kernel is math library, daemon is orchestrator. Pure functions don't have side effects.
4. **Don't put guard clauses (< 2 samples, < 60s, invalid load) in kernel** — they're policy (orchestrator), not physics.
5. **Don't make one monolithic `battery_math.py` file** — modularize by change frequency to enable focused evolution.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Trapezoidal integration for area-under-curve | Custom loop with Δt handling | stdlib `math` (manual trapz loop) | Voltage-SoC area calculation is 4 lines; hand-rolling risks off-by-one errors on boundary conditions. Reference: `soh_calculator.py` lines 76-81 already correct. |
| Variance / coefficient of variation | Custom variance implementation | `statistics.stdev()` + `mean()` | Confidence scoring needs population std (÷n, not ÷n-1). stdlib handles this correctly; hand-rolling risks Fisher-Bessel bias. |
| Peukert's law formula | Approximation or lookup table | Exact formula from physics | Formula `T = T_rated × (I_rated/I_actual)^n` is 2-line calculation. Approximation introduces ±5% error; exact is 3x simpler. Source: `runtime_calculator.py` lines 17-36. |
| Time-dependent weighting (age-based decay) | Global variable for "current time" | Inject `current_time` parameter | Global state breaks simulation (must mock time globally). Parameter injection = one-line function signature change. |
| LUT interpolation | Binary search + linear interpolation | Existing `bisect` + manual lerp | Already implemented and tested in `soc_predictor.py`. Hand-rolling duplicates code and diverges on edge cases. |

## Common Pitfalls

### Pitfall 1: Circular Dependency (Capacity ↔ Peukert)

**What goes wrong:** Capacity estimation depends on Peukert exponent (Q = ∫I×t × f(n)). Auto-calibrating both simultaneously causes oscillation with no convergence. Capacity estimate changes → Peukert updates → changes capacity → ...

**Why it happens:** Tempting to refine both from the same discharge data. But they measure different things (capacity vs. discharge nonlinearity); simultaneous refinement has no fixed point.

**How to avoid:** **Lock Peukert at 1.2 for v2.0 (VAL-02 constraint).** This is a design decision, not a limitation. Refinement is deferred to Phase 12.2+ (v2.1), after capacity estimation is proven stable.

**Warning signs:** If simulation shows capacity estimates oscillating (not converging within ±5%) after 10+ events, Peukert may be drifting. Check: is exponent auto-calibrating? If yes, stop it. Revert to fixed 1.2.

**Code guard:**
```python
# Phase 12.1 kernel: no exponent auto-calibration
class BatteryState:
    peukert_exponent: float  # Fixed at 1.2, never modified by formulas
    # (Only monitor.py can set it, and only to 1.2)
```

### Pitfall 2: Aggregate Stability Test Too Loose (±3% After 10 Iterations)

**What goes wrong:** Test declares "±1% input → ±3% after 10 iterations = stable" and passes. System actually amplifies ±1% by 1.1x per iteration (1% × 1.1^10 = 2.6% < 3%). But 1% × 1.1^100 = 13,780% → blowup over 100 iterations (3 months).

**Why it happens:** Confusing aggregate error tolerance with per-iteration amplification factor. Aggregate test is a smoke test only; per-iteration test is the real gate.

**How to avoid:** **Primary test is per-iteration Lyapunov: `divergence[i] / divergence[i-1] < 1.0` for all i > warmup_period (3).** This catches even slow accumulation (1.1x/iteration) early. Aggregate ±3% is secondary confirmation only.

**Warning signs:** If per-iteration ratio is 0.95 at iteration 5 but 1.05 at iteration 15, instability is emerging. Investigate: did SoH cross a threshold? Is load profile changing? Run adversarial scenario.

**Test structure (from expert panel):**
```python
def test_lyapunov_stability():
    """Per-iteration divergence ratio < 1.0 is the gate."""
    baseline_state = initial_battery_state()
    perturbed_state = apply_perturbation(baseline_state, delta=0.01)  # ±1% capacity

    for iteration in range(100):
        # Simulate one discharge event
        baseline_state = simulate_discharge(baseline_state, event=discharge_events[iteration])
        perturbed_state = simulate_discharge(perturbed_state, event=discharge_events[iteration])

        divergence_i = abs(perturbed_state.capacity_ah_measured - baseline_state.capacity_ah_measured)
        divergence_prev = (abs(divergence[i-1]) if i > 0 else 0.01)

        ratio = divergence_i / divergence_prev if divergence_prev > 0 else 1.0

        # Primary gate: ratio < 1.0 (divergence shrinking or stable)
        if iteration > 3:  # Warmup period
            assert ratio < 1.0, f"Iteration {iteration}: divergence amplified by {ratio}"

        divergence.append(divergence_i)
```

### Pitfall 3: Flicker Storm Pollutes SoH History

**What goes wrong:** 20 × 3-second power transitions in 1 hour. Each transition updates SoH with Bayesian weight ≈ 0.0035 (negligible signal). SoH history records 20 junk entries. Replacement predictor regression model now has 1000+ noise points degrading fit quality.

**Why it happens:** No minimum discharge duration filter. System accepts any OB→OL transition as valid discharge event.

**How to avoid:** **Minimum 30-second discharge for SoH update (VAL-01 enforcement).** Kernel function `calculate_soh_from_discharge()` returns `None` if duration < 30s. Orchestrator skips SoH update but still increments `cycle_count` and `cumulative_on_battery_sec` (they track frequency, not magnitude).

**Warning signs:** If SoH history entries group within seconds (not hours/days), flicker storm is happening. Check monitor.py event classifier — is it generating spurious discharge events?

**Code guard (Phase 12.1 kernel):**
```python
def calculate_soh_from_discharge(...) -> float | None:
    """Returns None if discharge < 30s (flicker storm), even if Bayesian weight is small."""
    discharge_duration = trimmed_t[-1] - trimmed_t[0]
    if discharge_duration < 30:
        logger.warning(f"Discharge {discharge_duration}s < 30s minimum; skipping SoH update")
        return None  # Don't pollute history
```

### Pitfall 4: Discharge Cooldown Missing (Flicker Treated as Two Discharges)

**What goes wrong:** Power line noise causes 300s discharge → 2s OL (capacitor kick-in) → 300s discharge continuation. System treats as 2 separate 300s discharges (two capacity estimates). Actually one 600s discharge (accurate capacity). Produces two half-accuracy measurements.

**Why it happens:** Monitor.py clears discharge buffer on any OL event without delay.

**How to avoid:** **Discharge cooldown: 60s OL before clearing buffer.** If OB→OL→OB within 60s, append to existing discharge buffer (continuation). Only clear after 60s OL confirmed (true power restoration).

**Warning signs:** Capacity estimates alternate high/low (two halves of same discharge). Check discharge events: are they systematically short? Look for OB/OL/OB patterns within minute of each other.

**Code guard (orchestrator only — Phase 12.1 kernel unchanged):**
```python
# In monitor.py _update_battery_health():
if self.state == EventType.OL and self.previous_state == EventType.OB:
    # Power loss detected, but don't clear buffer immediately
    self.discharge_buffer_clear_countdown = 60  # seconds

if self.state == EventType.OB:
    self.discharge_buffer_clear_countdown = None  # Power restored, wait out cooldown

if self.discharge_buffer_clear_countdown is not None:
    self.discharge_buffer_clear_countdown -= POLL_INTERVAL
    if self.discharge_buffer_clear_countdown <= 0:
        self.discharge_buffer.clear()  # Only clear after 60s OL confirmed
```

### Pitfall 5: Tightened Voltage Floor Not Enforced

**What goes wrong:** During OB discharge, UPS voltage sags from load. One corrupted NUT poll returns 8.5V (impossible, UPS cuts off at 10.5V). Sample is included in LUT interpolation. LUT gets calibration point (8.5V, SoC=low) that doesn't exist in reality. Future SoC lookups corrupted.

**Why it happens:** Global voltage bounds check (8.0V–15.0V) is too loose. 8.5V passes, but it's physically impossible during active discharge.

**How to avoid:** **Tighten discharge-mode voltage floor to 10.0V when OB.** Global check (8.0–15.0V) is fine for OL state (relaxed = no load). During OB state with active discharge, floor must be 10.5V (UPS cutoff).

**Warning signs:** LUT entries cluster at 8.5–9.5V (should never appear during OB). Check monitor.py event classifier: is it correctly tracking OB vs. OL state when updating LUT?

**Code guard (Phase 12.1 kernel):**
```python
def soc_from_voltage(voltage: float, lut: tuple, state: str = "OL") -> float:
    """Lookup SoC from voltage with state-dependent floor.

    Args:
        voltage: Measured voltage (V)
        lut: Lookup table (immutable tuple)
        state: "OL" (on-line, no load) or "OB" (on-battery, active discharge)

    Returns:
        SoC [0, 1] or None if voltage out of bounds
    """
    if state == "OB":
        floor = 10.0  # Discharge mode: UPS can't go below this
    else:
        floor = 8.0   # Global floor: worst-case (battery dying)

    if voltage < floor or voltage > 15.0:
        logger.warning(f"Voltage {voltage}V out of bounds [{floor}, 15]")
        return None  # Skip corrupted sample

    # Rest: bisect + lerp on LUT
```

### Kernel Function: Pure Peukert Calculation

**Source:** `src/runtime_calculator.py` lines 17-36 (existing, already pure)

```python
def peukert_runtime_hours(
    load_percent: float,
    capacity_ah: float = 7.2,
    peukert_exponent: float = 1.2,
    nominal_voltage: float = 12.0,
    nominal_power_watts: float = 425.0
) -> float:
    """Core Peukert calculation: full-charge runtime in hours at given load.

    Pure function: no time.time(), no I/O, no side effects.

    Args:
        load_percent: Current load [0, 100] %
        capacity_ah: Battery capacity (Ah)
        peukert_exponent: Peukert exponent n [1.0, 1.4]; fixed at 1.2 for v2.0
        nominal_voltage: UPS nominal voltage (12V for VRLA)
        nominal_power_watts: UPS nominal power (425W for UT850EG)

    Returns:
        Runtime in hours at SoC=1.0, SoH=1.0 (caller scales by SoC/SoH)
        Returns 0.0 if load_percent <= 0

    Physics:
        T = T_rated × (I_rated / I_actual)^n
        where T_rated = capacity_ah / I_rated = 20 hours (C/20 discharge rate)
              I_rated = capacity_ah / 20
              I_actual = (load_percent / 100) × (power_watts / voltage)
    """
    if load_percent <= 0:
        return 0.0

    I_rated = capacity_ah / 20.0
    I_actual = load_percent / 100.0 * nominal_power_watts / nominal_voltage
    T_rated = capacity_ah / I_rated  # = 20 hours
    return T_rated * (I_rated / I_actual) ** peukert_exponent
```

### Kernel Function: SoH with Time Injection

**Source:** `src/soh_calculator.py` refactored for Phase 12.1

```python
def _weighted_average_by_voltage(
    lut_entries: List[Dict],
    current_time: float = None,  # INJECTED: enables simulation without mocking
    decay_hours: float = 24 * 30  # 30 days
) -> float:
    """Weighted average voltage with age-based decay.

    LUT entries older than decay_hours get lower weight, eventually 0.
    This prevents stale calibration points from biasing SoC lookups.

    Args:
        lut_entries: List of {"voltage": V, "timestamp": t_unix, "source": str}
        current_time: Current timestamp (seconds since epoch).
                     If None, uses time.time() (daemon behavior).
                     If provided, uses that value (simulator behavior).
        decay_hours: Age beyond which weight → 0

    Returns:
        Weighted average voltage [V]
    """
    if current_time is None:
        current_time = time.time()  # Default: real time

    if not lut_entries:
        return 12.0  # Default nominal voltage

    weights = []
    voltages = []

    decay_seconds = decay_hours * 3600

    for entry in lut_entries:
        v = entry['voltage']
        t_unix = entry['timestamp']
        age_seconds = current_time - t_unix

        if age_seconds < 0:  # Clock jump (NTP correction)
            weight = 0.0
        elif age_seconds > decay_seconds:
            weight = 0.0
        else:
            # Linear decay: weight = 1.0 at age=0, 0.0 at age=decay_seconds
            weight = 1.0 - (age_seconds / decay_seconds)

        if weight > 0:
            weights.append(weight)
            voltages.append(v)

    if not weights:
        return 12.0  # No recent entries; return nominal

    total_weight = sum(weights)
    return sum(v * w for v, w in zip(voltages, weights)) / total_weight
```

                load_percent=load_percent
            )

            if new_soh is not None:
                baseline_state = replace(baseline_state, soh=new_soh)
                baseline_state = replace(
                    baseline_state,
                    cycle_count=baseline_state.cycle_count + 1,
                    cumulative_on_battery_sec=baseline_state.cumulative_on_battery_sec + discharge_duration
                )

            # Simulate same discharge on perturbed state
            new_soh_perturbed = calculate_soh_from_discharge(
                v_series, t_series,  # Same event
                reference_soh=perturbed_state.soh,
                capacity_ah=perturbed_state.capacity_ah_rated,  # Different capacity
                load_percent=load_percent
            )

            if new_soh_perturbed is not None:
                perturbed_state = replace(perturbed_state, soh=new_soh_perturbed)
                perturbed_state = replace(
                    perturbed_state,
                    cycle_count=perturbed_state.cycle_count + 1
                )

            # Compute divergence
            divergence = abs(
                perturbed_state.soh - baseline_state.soh
            )
            divergence_history.append(divergence)

            # Check per-iteration Lyapunov ratio
            if len(divergence_history) > 3:  # Warmup: skip first 3
                ratio = divergence_history[-1] / divergence_history[-2]
                assert ratio < 1.0, \
                    f"Unstable at day={day}, event={event_idx}: ratio={ratio:.3f} > 1.0"

    # Secondary smoke test: aggregate divergence < ±3%
    final_divergence = divergence_history[-1]
    assert final_divergence < 0.03, \
        f"Final divergence {final_divergence:.4f} > 3%"

def test_year_simulation_permutation_invariance():
    """Path-independent: 10 events in 5 random orderings agree within ±2%."""
    # Generate 10 discharge events (fixed)

    for shuffle_idx in range(5):
        random.seed(42 + shuffle_idx)  # Different seed per shuffle
        shuffled_events = random.sample(events, len(events))  # Reorder

## State of the Art

### Peukert's Law (Stable, 1897)

| Aspect | Status | Impact |
|--------|--------|--------|
| Formula | Established physics, 127 years old | No changes needed; fixed at 1.2 for v2.0 |
| Parameter identification | From discharge curves | Deferred to Phase 12.2+ (v2.1) when capacity is stable |
| Applicability to VRLA | High (±3% typical) | Sufficient for runtime prediction; narrower band than lead-acid general (±5%) |

The exponent `n` varies by chemistry: LiPo ~1.0, VRLA ~1.2, NiCd ~0.9. CyberPower UT850EG is lead-acid (VRLA), so 1.2 is well-grounded. No update needed for decades.

### SoH Calculation (Evolving, 2025–2026)

| Update | Date | Change | Status |
|--------|------|--------|--------|
| Initial trapezoidal integration | 2026-03-14 | Area-under-curve formula | Proven in 205 tests, v1.1 shipped |
| Bayesian duration weighting | 2026-03-14 | Weight short discharges lower (0.001–1.0) | Reduces bias on flicker events, Phase 11 |
| 30-second minimum for SoH update | 2026-03-15 | Returns None if < 30s | Prevents flicker-storm history pollution (VAL-01) |
| Age-decay for LUT weighting | 2025-2026 | Exponential / linear decay on age | Prevents stale calibration from biasing SoC |

The approach is crystallizing: voltage-area analysis is reliable. Refinements are around data quality (minimum duration, age decay) rather than formula changes.

### Validation Approach (New, 2026)

| Technique | Origin | Application to Phase 12.1 |
|-----------|--------|---------------------------|
| Lyapunov per-iteration stability | Dynamical systems theory | Primary gate: divergence[i]/divergence[i-1] < 1.0 |
| Permutation invariance test | Machine learning (model bias detection) | Catches path-dependent bias in Bayesian SoH |
| Fixed-point convergence | Numerical analysis | 20 identical discharges → converge within 1% |
| Adversarial scenarios | Software testing | Flicker storm, interrupted discharge, voltage spike, clock jump |

These are new to battery monitoring but standard in control systems. Applying them to Phase 12.1 is disciplined validation, not overkill.

### Deprecated / Outdated

- **Coulomb counting without voltage anchor** — Drifts ±30% over time. Replaced by voltage-anchored method (Phase 12).
- **Global voltage bounds [8.0, 15.0]V** — Too loose during discharge. Tightened to [10.0, 15.0]V during OB state.
- **EMA-filtered load in discharge buffer** — Lags true load; degrades coulomb accuracy. Replaced with raw load (VAL-01).
- **Auto-calibrating Peukert during capacity estimation** — Circular dependency. Deferred to Phase 12.2+.

## Open Questions

1. **Simulation Time Budget**
   - What we know: "< 30 seconds for 365 days of synthetic events" is the target. 365 × 3 events/week ≈ 1000 synthetic discharge iterations.
   - What's unclear: What granularity per event (10 samples? 100?) and what trade-off between simulation speed and fidelity.
   - Recommendation: Start with 10 voltage samples per synthetic discharge (enough to compute area-under-curve). If simulations timeout, reduce to 3-5. If too coarse, increase to 100 and profile.

2. **Seasonal Thermal Variation**
   - What we know: VRLA capacity varies ±5% over ±10°C range. Winter vs. summer estimates look like degradation if not documented.
   - What's unclear: Should simulation include synthetic thermal variation? Or just document as known limitation?
   - Recommendation: Phase 12.1 simulation doesn't include thermal variation (no sensor in v2.0). Phase 12.1 success criteria note that if capacity estimates vary seasonally within CoV < 10%, system is working as designed. Defer thermal modeling to v3.0.

3. **LUT Mutation in Orchestrator**
   - What we know: `interpolate_cliff_region` mutates LUT to smooth near-cutoff region (Phase 11, `soh_calculator.py`). Must stay in orchestrator, not kernel.
   - What's unclear: Is LUT a pure `lut_in → lut_out` transform, or does orchestrator own the mutation entirely?
   - Recommendation: Define `interpolate_cliff_region` as `(lut: tuple) → tuple` in kernel (pure), but move implementation to `monitor.py` for now (orchestrator responsibility). Kernel imports it but doesn't modify LUT state. Revisit if formula needs frequent tuning.

4. **Discharge Cooldown Persistence**
   - What we know: If OB→OL→OB within 60s, treat as continuation (don't clear buffer). This prevents power flicker from creating spurious two-half-discharge measurements.
   - What's unclear: How to track cooldown state across polls (10-second interval)? Does `self.discharge_buffer_clear_countdown` persist in model.json?
   - Recommendation: Cooldown counter is in-memory (Daemon runtime state), not persistent. Worst case: power cuts out during cooldown, daemon restarts, loses buffer. Acceptable: next full discharge updates estimate anyway.
