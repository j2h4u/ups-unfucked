**Part A: test_monitor_integration.py**

The `mock_daemon` fixture (line 57-65) already creates a REAL MonitorDaemon with real SagTracker, SchedulerManager, DischargeCollector. The `TestPollOnceCallChain` class is the gold standard — it only mocks `_update_battery_health` (complex subsystem boundary) and `_write_calibration_points` (disk I/O).

Specific anti-patterns to fix:

1. **Line 390: `daemon.capacity_estimator = MagicMock()`** in `test_handle_discharge_complete_logging` within `TestDischargeEventLogging` class.

   This replaces the real CapacityEstimator with a MagicMock. The test needs to control what `estimate()` and `has_converged()` return. Since CapacityEstimator is a domain object (not I/O), we should use the real one BUT the test needs deterministic output.

   Strategy: Keep the MagicMock for `capacity_estimator` ONLY in tests that need to control estimation output (this is a contract boundary, not I/O boundary). However, add a comment explaining why:
   ```python
   # CapacityEstimator mocked to control estimation output — not I/O, but
   # test needs deterministic capacity values. Real estimator tested in
   # test_capacity_estimator.py with Monte Carlo validation.
   daemon.capacity_estimator = MagicMock()
   daemon.discharge_handler.capacity_estimator = daemon.capacity_estimator
   ```

2. **Line 761: `daemon.discharge_collector._write_calibration_points.call_count`** in `TestPollOnceCallChain.test_ob_accumulates_samples`.

   This asserts call_count on a mocked private method. `_write_calibration_points` is disk I/O (writes to model.json via safe_save), so mocking it is correct. But the assertion should verify the outcome, not the call count.

   Replace:
   ```python
   # BEFORE:
   assert daemon.discharge_collector._write_calibration_points.call_count == len(ob_voltages)
   # AFTER:
   # Calibration points are written to disk on each OB poll. Since _write_calibration_points
   # is mocked (disk I/O), verify discharge buffer has the expected samples instead.
   assert len(daemon.discharge_collector.discharge_buffer.voltages) == len(ob_voltages), \
       f"Expected {len(ob_voltages)} voltage samples accumulated, got {len(daemon.discharge_collector.discharge_buffer.voltages)}"
   ```

3. **Line 783: `daemon._update_battery_health.assert_called()`** in `TestPollOnceCallChain.test_full_blackout_cycle`.

   This is an acceptable mock — `_update_battery_health` is explicitly mocked as a complex subsystem boundary (the fixture comment says so). The mock side_effect already clears the discharge buffer. However, we can strengthen the assertion to verify the observable EFFECT rather than just "it was called":
   ```python
   # BEFORE:
   daemon._update_battery_health.assert_called()
   # AFTER — verify the effect of _update_battery_health (buffer was cleared by side_effect):
   assert not daemon.discharge_collector.discharge_buffer.collecting, \
       "EVT-05: discharge buffer should be cleared after OB->OL health update"
   assert len(daemon.discharge_collector.discharge_buffer.voltages) == 0, \
       "EVT-05: discharge buffer voltages should be empty after health update"
   ```

**Part B: test_monitor.py — TestCapacityEstimatorIntegration class (lines 1084-1320)**

The tests in this class use `daemon.capacity_estimator = MagicMock()` (lines 1113, 1195, 1234, etc.) to control estimation results. Since these tests verify the orchestrator's handling of estimator output (not the estimator itself), this is a valid boundary mock. However, improve the assertions:

