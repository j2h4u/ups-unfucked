### CAP-05 Gap Details

**What works:** `--new-battery` flag triggers `_reset_battery_baseline()` correctly — clears capacity estimates, adds fresh SoH entry, resets cycle count. The functional behavior is correct.

**What's broken:** `src/monitor.py` line 347-348 clears `new_battery_requested` flag immediately after processing, before `model.save()` can persist `True`. Two tests fail:
- `test_new_battery_flag_true`: expects flag to be True after daemon init
- `test_new_battery_flag_persistence`: expects flag persisted to model.json as True

**Impact:** The flag is one-shot (set→act→clear), so the reset works. But the test contract says the flag should be readable after init. This is a test vs implementation semantic mismatch introduced when Phase 13 added the reset logic.

**Fix:** Either update tests to match one-shot semantics, or defer flag clearing to after model state is readable.
